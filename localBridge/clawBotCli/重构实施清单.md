# clawBotCli 重构实施清单

> 目标：将 `localBridge/clawBotCli` 从“测试脚本集合”重构为“可复制到其它工程中的 Python 类库”，让 AI 或开发者可以基于这个类库继续开发新功能。
>
> 核心约束：**最终产物必须是类库，而不是一组测试脚本**。复制到其它项目后，应可直接 `import` 使用，并具备清晰的基础能力层、组合能力层与示例层。

---

## 一、重构目标

### 1. 最终目标

将当前目录整理为一个可嵌入其它工程的 Python 包，满足以下要求：

1. 可复制到其它工程目录中直接使用
2. 可通过统一入口类进行调用
3. 对外暴露稳定、清晰、可组合的 Python API
4. 屏蔽底层复杂的 Twitter/X 原始 GraphQL 响应结构
5. 支持 AI 基于该类库继续开发新功能
6. 测试、示例、脚本、类库主体彻底分层

### 2. 非目标

以下内容不是这次重构的最终产物：

- 继续扩展 `tests/` 目录作为主要能力出口
- 让用户通过复制测试脚本来使用能力
- 让外部工程直接依赖 `utils/api_client.py` 里的混合式接口
- 继续把 workflow、测试、HTTP 客户端、解析逻辑混在同一个文件或目录中

---

## 二、现状问题总结

当前 `localBridge/clawBotCli` 主要存在以下问题：

1. **职责混杂**：
   - `utils/api_client.py` 同时承担 X API、AI API、tab 控制、媒体上传任务封装
   - `tests/` 同时承担测试、示例、workflow、人工验证脚本角色

2. **返回结构过于底层**：
   - 读取接口返回大量原始 Twitter/X GraphQL 嵌套结构
   - 用户或 AI 需要手动解析 `instructions / entries / tweet_results / legacy` 等字段

3. **组合逻辑分散在测试脚本中**：
   - 如提取 tweet_id、提取 pinned tweet、搜索后提取 screen_name 等逻辑散落在 `tests/*.py` 中
   - 不利于沉淀成真正可复用的类库能力

4. **目录角色不清晰**：
   - `openclaw.py` 更像 demo/workflow，而不是类库核心文件
   - `response_parser.py` 更像测试辅助工具，而不是正式领域解析层

5. **不利于 AI 扩展**：
   - 命名偏测试脚本风格
   - 缺少统一入口、缺少结构化模型、缺少领域分层

## 四、各层职责定义

### 1. transport 层

职责：
- 只负责 REST API 映射
- 组织请求参数、处理 HTTP 错误、返回原始 JSON/bytes

不负责：
- 业务流程
- tweet/user 结构解析
- 工作流编排

建议提供：

- `XApiTransport`
- `AIApiTransport`
- `TaskApiTransport`

### 2. domain 层

职责：
- 将原始响应解析成稳定、结构化的数据模型
- 屏蔽复杂嵌套 JSON 细节

建议定义的数据模型：

- `XStatus`
- `XTab`
- `XUser`
- `XTweet`
- `ActionResult`
- `AIPlatformStatus`
- `AIMessageResult`
- `MediaUploadResult`

要求：
- 保留 `raw` 字段，便于 AI 或高级调用方继续扩展

### 3. services 层

职责：
- 向外提供“基础能力层”
- 调用 transport + domain
- 返回结构化对象，而不是原始 JSON

这是未来用户和 AI 最常用的层。

### 4. workflows 层

职责：
- 封装常见的多步骤组合能力
- 提供高层积木给 AI 或外部工程使用

例如：
- 获取置顶推文并回复
- 读取时间线第一条推文并互动
- 搜索用户并读取资料
- 上传媒体并发帖

### 5. examples / scripts / tests

职责分离：
- `examples/`：给用户参考的示例脚本
- `scripts/`：命令行 demo 或工具脚本
- `tests/`：真正自动化测试

## 六、对外 API 设计蓝图

### 1. 统一入口类

建议对外暴露：

```python
from clawbot import ClawBotClient

client = ClawBotClient(base_url="http://127.0.0.1:10088")
```

然后通过子域组织能力：

```python
client.x
client.ai
client.media
client.workflows
```

---

#### `client.x.actions`

- `retweet(tweet_id, tab_id=None) -> ActionResult`

## 七、关键设计原则

### 1. raw 与 parsed 双通道并存

例如：
- `get_timeline_raw()`
- `list_tweets()`

这样既方便高级扩展，也保证普通调用易用。

### 2. 领域对象统一为 dataclass

推荐对高频对象统一建模，如：
- `XTweet`
- `XUser`
- `XStatus`
- `ActionResult`

### 3. 统一异常体系

新增：
- `ClawBotError`
- `ApiRequestError`
- `ParseError`
- `AuthenticationError`
- `TaskTimeoutError`
- `MediaUploadError`

### 4. 隐藏底层复杂结构

不要让外部用户或 AI 直接依赖：
- `instructions`
- `entries`
- `itemContent`
- `tweet_results`
- `legacy`

这些都应被 parser 层吸收。

### 5. 命名按领域组织

避免继续使用测试风格命名，如：
- `test_basic_read`
- `test_workflow`
- `response_parser`

改为：
- `timeline`
- `tweets`
- `users`
- `actions`
- `search`
- `media`

## 八、分阶段实施计划

### Phase 1：建立类库骨架

目标：建立一个可 import 的基础类库结构。

任务：
1. 新建 `clawbot/` 目录
2. 新建 `__init__.py`
3. 新建 `client.py`
4. 新建 `errors.py`
5. 新建子目录：
   - `transport/`
   - `domain/`
   - `services/`
   - `workflows/`
   - `upload/`

交付结果：
- 项目已具备类库基础目录
- 可开始迁移现有逻辑

### Phase 2：迁移 transport 层

目标：将 `APIClient` 拆分为原始 REST 映射层。

任务：
1. 抽取基础请求类 `BaseApiTransport`
2. 迁移 X 相关请求到 `transport/x_api.py`
3. 迁移 AI 相关请求到 `transport/ai_api.py`
4. 迁移 task 相关请求到 `transport/task_api.py`

交付结果：
- 原始 HTTP 能力不再集中在一个混合文件里

### Phase 3：建立 domain 模型与 parser

目标：将 raw response 变成可用的结构化领域对象。

任务：
1. 定义核心 dataclass
2. 从现有测试脚本中抽取解析逻辑
3. 实现：
   - timeline parser
   - tweet parser
   - user parser
   - pinned tweet parser
   - search parser

交付结果：
- 绝大部分调用不再依赖底层嵌套 JSON

### Phase 4：实现 services 层

目标：形成可供外部工程直接调用的基础能力层。

任务：
1. 实现 `x_status.py`
2. 实现 `x_read.py`
3. 实现 `x_actions.py`
4. 实现 `x_tabs.py`
5. 实现 `ai_chat.py`
6. 实现 `media.py`

交付结果：
- 用户可通过结构化服务调用基础能力

### Phase 5：抽离 workflow 能力

目标：把散落在 `tests/` 与 `openclaw.py` 中的组合逻辑沉淀为 workflow。

任务：
1. 抽取 timeline 读取+互动工作流
2. 抽取 search + profile 工作流
3. 抽取 pinned tweet + AI reply 工作流
4. 抽取媒体上传 + 发帖工作流

交付结果：
- 高价值组合能力可直接复用

### Phase 6：整理 examples / tests / scripts

目标：完成目录角色分离，形成可维护结构。

任务：
1. 将 demo 脚本迁移到 `examples/` 或 `scripts/`
2. 将原有 `tests/` 分类迁移为 unit/integration
3. 补充面向类库使用者的 README

交付结果：
- 项目从脚本集合正式转为类库项目

1. `client.x.status.is_logged_in()`
2. `client.x.timeline.list_tweets()`

## 十、实施注意事项

1. `navigate_tab(path)` 的语义需要明确，后续应区分：
   - 导航到站内 route
   - 导航到用户主页

2. 当前 `MediaUploadTask.upload_video()` 命名过窄，建议改为：
   - `upload_media()` 或 `upload_file()`

3. `APIClient._request()` 不应继续作为对外依赖点，需补全正式公开方法

4. 类库内必须尽量减少测试语义命名，避免误导后续 AI 开发

## 十三、验收标准

重构完成后，应满足以下验收条件：

1. 外部工程可以直接复制 `clawbot/` 目录并导入使用
2. 存在统一入口：`from clawbot import ClawBotClient`
3. 用户无需直接解析 Twitter/X 原始响应嵌套结构
4. 常见基础能力可通过结构化 service 调用
5. 常见组合能力可通过 workflow 调用
6. 示例、脚本、测试与类库本体分离
7. AI 能够基于类库结构快速新增功能，而不是继续复制测试脚本

---

## 十四、一句话总结

本次重构的核心，不是“把测试脚本整理得更好看”，而是把 `localBridge/clawBotCli` 正式升级为一个**可复制、可导入、可扩展、适合 AI 二次开发的 Python 类库**。
