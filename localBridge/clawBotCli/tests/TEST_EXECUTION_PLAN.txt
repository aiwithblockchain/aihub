ClawBot CLI Manual Test Plan
============================

Purpose
-------
This plan is for slow, manual, one-test-at-a-time execution. After each executed test, stop and wait for the user's next command.

Working rules
-------------
1. Execute only one test item at a time.
2. After each test:
   - record result
   - patch code or test if needed
   - mark the item status
   - stop and wait for the next command
3. Prefer read flows first, then write flows.
4. Write tests must use clearly labeled test content so the resulting X actions are easy to identify.
5. Media tests may use files under `test_media/`.
6. For multi-instance environments, prefer explicit `instanceId` when the script supports it.

Status legend
-------------
- [ ] not started
- [~] in progress
- [x] done
- [!] blocked / needs script update

Patch expectations before execution
-----------------------------------
Some scripts are older than the latest API design. Before or after each test, check whether the script needs a surgical update.

High-priority likely updates
----------------------------
1. `tests/test_read_apis.py`
   - likely needs explicit `instanceId` support for multi-instance setups
   - may need better verification that `get_tweet()` returns the focal tweet for the requested ID
2. `tests/test_tweet_details.py`
   - should verify the latest canonical logic more explicitly
   - should prefer checking focal tweet match, nested reply extraction, and latest raw-vs-structured semantics
3. `tests/test_write_apis.py`
   - currently interactive and broad
   - may need splitting or flags so we can run one action at a time safely
4. any live write script
   - may need a common way to inject `instanceId`
   - may need clearer logging of created tweet IDs for later cleanup
