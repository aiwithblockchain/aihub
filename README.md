# AIHub

AIHub 是一个面向 AI Agent 的本地工具中间层，也可以理解为一个 AI tool hub / router。

它的目标很直接：让 `clawbot` 不必在每次执行任务时都去消化大量页面上下文，而是通过 AIHub 获取更直接的结构化结果，再把结构化指令分发到具体应用执行端。

## 项目背景

当前 `clawbot` 在执行网页任务时，通常会依赖浏览器侧的信息输入，例如：

- 页面截图
- 页面快照或页面结构信息
- 基于这些上下文再决定后续动作

这种方式并不意味着它“只能读取截图或完整 DOM”。实际上，`clawbot` 已经具备更结构化的浏览器交互能力。但在登录态网站、复杂业务页面、需要高稳定性的操作场景中，它仍然往往需要先理解较多与任务有关或无关的页面信息，才可能完成具体动作。

这会带来几个典型问题：

- 延迟更高：需要先观察页面，再推理，再执行
- 成本更高：上下文更大，token 消耗更多
- 准确率下降：无关信息越多，误判概率越高
- 工程稳定性更差：页面变化会放大模型决策的不确定性

AIHub 要解决的，就是这层“从页面理解到任务执行”的额外损耗。

## 核心思路

AIHub 位于 `clawbot` 与具体应用之间，负责把面向页面的泛化分析，收敛成面向任务的结构化协议。

它的职责包括：

- 接收 `clawbot` 发来的结构化任务
- 将任务路由到对应应用的执行通道
- 把执行端返回的数据整理为结构化结果
- 统一对外暴露可调用的接口和协议文档

最终效果是：

- `clawbot` 更少关注页面细节
- 应用执行端更专注于完成具体动作
- 整体链路更快、更便宜、更稳定

## 架构方案

AIHub 当前的方案由两部分组成。

### 本地 WebSocket 服务

AIHub 会启动一个本地运行的 WebSocket 服务器，用来建立实时消息通道。

这个通道主要负责：

- 创建频道
- 为不同应用分配独立通道
- 维护 `clawbot` 与应用执行端之间的结构化消息流

这样，`clawbot` 可以按应用或能力把任务投递到对应频道，而不需要每次都从页面原始信息重新理解一遍。

### 面向 clawbot 的 REST API

除了 WebSocket 通道，AIHub 还会暴露一组 REST API 给 `clawbot` 调用。

这些 API 会配套说明文档，明确：

- 每个接口的用途
- 请求方式
- 请求参数格式
- 返回数据格式
- 错误返回格式

## 示例：`tweetClaw`

当前仓库以 `clawbot` 运营一个 Twitter / X 账号作为示例。

其中 `tweetClaw` 是一个专用浏览器扩展，它负责：

- 与浏览器运行环境交互
- 模拟人对网页的实际操作
- 读取登录后的页面信息
- 执行具体的 Twitter 操作

当前的职责边界是：

- 登录动作默认由人手动完成
- 登录完成后，由 `tweetClaw` 在该浏览器会话中继续执行任务

`tweetClaw` 负责的典型操作包括：

- 获取推文
- 查询内容
- 点赞
- 转发
- 发推

## 数据流

这个架构中的分工如下：

- `clawbot`：理解主人的意图，并生成结构化任务
- `AIHub`：作为中间层接收、路由、汇总任务结果
- `tweetClaw`：在浏览器中执行真实网页操作，并返回结构化结果

一次典型调用链路如下：

1. 用户向 `clawbot` 发出任务，例如查询账号内容、点赞、转发或发推。
2. `clawbot` 将意图转换成结构化请求，发送给 AIHub。
3. AIHub 根据目标应用和频道配置，将请求路由到对应执行端。
4. `tweetClaw` 在已登录的浏览器环境中执行具体操作。
5. 执行结果以结构化数据形式返回给 AIHub。
6. AIHub 再将结果返回给 `clawbot`。

整个过程中，核心原则是：

- 尽量不传递无关的大量页面原始信息
- 尽量只传递任务真正需要的结构化数据

## 仓库结构

- `localBridge/`：本地桥接相关实现或平台侧说明
- `tweetClaw/`：浏览器扩展示例，演示如何承接 AIHub 路由来的结构化任务
- `docs/`：后续用于补充接口文档、协议说明和使用指南

## 安全假设

当前 AIHub 默认监听本地 IP / 本地端口，不包含额外鉴权机制。

当前版本采用的默认前提是：

- 本地回环或本地网络环境可信
- 调用本地服务端口的客户端是受控的

因此，它更适合本地开发、调试和单机运行场景。

如果未来需要跨设备、跨网络或多租户使用，则需要补充：

- 鉴权机制
- 权限控制
- 请求签名或 token 校验
- 更严格的网络暴露策略

## 项目目标

AIHub 希望把“AI 理解意图”和“应用执行动作”之间的链路标准化。

对于 `clawbot`，理想状态不是继续“看页面猜操作”，而是像调用工具一样，直接调用面向任务的结构化接口。对于具体应用，理想状态也不是暴露大量原始页面信息，而是暴露稳定、可执行、可返回结果的能力端口。

最终，AI 与真实应用之间的交互方式，将从“基于页面理解的弱耦合自动化”，升级为“基于结构化协议的直接协作”。
