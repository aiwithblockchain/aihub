## 1. 文档目标

本文档将以下三份设计文档收敛为可执行的工程任务拆解：

- `0.需求说明.md`
- `1.技术架构说明.md`
- `2.协议规范说明.md`

本文档关注：

- 实现顺序
- 模块拆分
- 任务依赖
- 每个任务包的交付物
- 最小验收标准

本文档不重复协议细节定义，只将其转化为执行任务。

---

## 2. 总体实施策略

AIHub v1.0 建议按以下顺序实施：

1. 先打底层骨架
2. 再做服务定义与生命周期管理
3. 再做运行时连接与路由调用
4. 再做对外 REST / MCP 入口
5. 再做能力说明与调试能力
6. 最后做联调、异常场景和文档收口

原因：

- 生命周期管理和服务定义是路由与调用的前提
- 运行时调用依赖服务注册、实例状态和错误模型
- 对外接口应建立在内部路由与控制协议稳定之后

## 3. 阶段划分

### 阶段 0：项目骨架与基础约束

#### 目标
建立 AIHub v1.0 的实现骨架，固定目录结构、配置入口、日志与错误基础设施。

#### 任务
- 建立 AIHub 服务端项目结构
- 建立扩展安装目录和运行目录约定
- 建立基础配置加载机制
- 建立统一日志接口
- 建立统一错误码常量和状态枚举常量

#### 交付物
- AIHub 主程序可启动
- 基础配置文件和默认目录结构可用
- 日志和错误码模块可被其他模块复用

#### 验收标准
- 启动时能输出版本、配置和目录初始化日志
- 错误码和状态枚举已有统一定义，后续模块不再重复发明

### 阶段 1：扩展包与 manifest 管理

#### 目标
先实现“已安装服务定义”的持久化管理事实，为后续注册、查询和路由提供基础。

#### 任务
- 实现扩展安装目录扫描器
- 实现 `manifest.json` 解析与校验器
- 实现路径解析规则
- 实现 `capabilities.json` 索引加载
- 实现已安装服务定义持久化视图
- 实现启用/禁用状态持久化视图

#### 交付物
- 服务安装记录模型
- manifest 校验模块
- capabilities 文档加载模块
- 启动恢复模块

#### 关键实现点
- 相对路径统一按扩展包安装根目录解析
- `update` 字段按可选字段处理
- AIHub 重启后必须能重建服务定义和能力说明视图

#### 验收标准
- 放入合法扩展包后，AIHub 能识别服务定义
- 非法 manifest 会被拒绝
- 重启后仍能看到相同的已安装服务列表和启用状态

### 阶段 2：扩展生命周期管理

#### 目标
实现安装、启用、禁用、手动检查更新、手动更新、卸载的完整基础生命周期。

#### 任务
- 实现安装逻辑
- 实现启用逻辑
- 实现禁用逻辑
- 实现本地更新源检查逻辑
- 实现本地更新包替换逻辑
- 实现卸载逻辑
- 实现生命周期状态机和状态冲突矩阵

#### 交付物
- Extension Lifecycle Manager
- 生命周期状态机
- 更新源解析器
- 生命周期管理 REST 接口

#### 关键实现点
- `installStatus`、`enableStatus`、`lifecycleStatus` 必须独立存储
- 所有生命周期操作必须经过状态冲突矩阵校验
- 更新和卸载前必须进入 `draining`

#### 验收标准
- 生命周期接口在允许状态下成功，在冲突状态下返回预期 `409`
- 本地更新源存在新版本时能被正确识别
- 卸载后服务定义、能力说明和安装内容被清理

### 阶段 3：服务注册表与运行时状态管理

#### 目标
实现“运行时事实”的统一状态层，支撑后续连接、心跳、路由和调试查询。

#### 任务
- 实现 Service Registry
- 实现实例状态模型
- 实现运行时内存注册表
- 实现 `lastSeen` 更新逻辑
- 实现实例并发状态和队列状态存储
- 实现 invocation 记录表

#### 交付物
- Service Registry / Instance Manager
- 运行时状态存储模块
- 调试查询数据接口

#### 关键实现点
- 持久化管理事实和运行时内存状态必须分离
- `lastSeen` 一律使用 AIHub 服务器收包时间
- `offline` 与 `unavailable` 必须显式区分

#### 验收标准
- 注册、心跳、断连都能正确反映到实例状态
- 服务状态查询和实例状态查询结果稳定可复现

### 阶段 4：内部 WebSocket Gateway

#### 目标
实现 AIHub 与子服务之间的唯一运行时连接通道。

#### 任务
- 实现内部 WebSocket 监听端点
- 实现连接建立和注册超时逻辑
- 实现连接到实例的映射关系
- 实现连接关闭清理逻辑
- 实现 WebSocket Ping/Pong 驱动

#### 交付物
- Service Connection Gateway
- 连接管理器
- 连接关闭与资源清理逻辑

#### 关键实现点
- 未完成 `service.register` 的连接必须被关闭
- 同一 `serviceId + instanceId` 重复连接必须按协议拒绝
- Ping/Pong 超时要将实例标记为 `offline`

#### 验收标准
- 合法子服务可以接入
- 非法注册会被拒绝并断开
- 异常断连后实例状态会被摘除

### 阶段 5：内部 JSON-RPC 控制协议

#### 目标
将 WebSocket 通道收敛为稳定的内部控制协议。

#### 任务
- 实现 JSON-RPC 编解码器
- 实现顶层错误码数值映射
- 实现 `service.register`
- 实现 `service.unregister`
- 实现 `service.heartbeat`
- 实现 `invoke`
- 实现 `invoke.cancel`

#### 交付物
- JSON-RPC 协议层
- 内部消息分发器
- 注册/心跳/调用处理器

#### 关键实现点
- 顶层 `error.code` 必须使用整数
- 字符串错误名放入 `error.data.code`
- `service.register` 和 `service.heartbeat` 需要按字段约束做校验

#### 验收标准
- 内部协议消息能被正确解析和处理
- 错误响应符合协议文档
- 下游业务错误不会冒充为顶层协议错误

### 阶段 6：Invocation 生命周期与路由调度

#### 目标
完成一次调用从进入 AIHub 到回传调用方的完整闭环。

#### 任务
- 实现 `invocationId` 生成器
- 实现挂起调用表
- 实现超时控制器
- 实现 `least_inflight` 实例选择策略
- 实现实例并发与队列控制
- 实现 `draining` 状态下的新请求拒绝与旧请求排空
- 实现 router-level / transport-level / downstream error 的分层返回

#### 交付物
- Router Core
- Invocation Lifecycle Manager
- 调度器
- 超时与取消模块

#### 关键实现点
- `draining` 下拒绝新请求，但允许挂起请求在窗口内完成
- timeout 与 cancel 必须联动 `invoke.cancel`
- `downstream_error` 必须走结果通道而非顶层错误通道

#### 验收标准
- 指定 `serviceId` 的请求能路由到正确实例
- 指定 `instanceId` 时能直达目标实例
- 超时、无实例、服务禁用、协议错误等场景都能给出正确分层错误

### 阶段 7：对外 REST API

#### 目标
把内部能力暴露为统一的对外主入口。

#### 任务
- 实现服务查询接口
- 实现实例查询接口
- 实现能力说明查询接口
- 实现调用接口
- 实现 invocation 调试查询接口
- 实现扩展包生命周期接口
- 实现 HTTP 状态码矩阵

#### 交付物
- Northbound REST API
- REST 错误响应层
- REST 请求校验层

#### 关键实现点
- 顶层 `data` / `error` 响应结构必须统一
- `GET /api/v1/services/{serviceId}/capabilities` 必须不依赖服务在线
- `GET /api/v1/invocations/{invocationId}` 只返回调用元数据

#### 验收标准
- 文档中定义的所有 REST 端点都可用
- HTTP 状态码和 JSON 响应结构一致
- 生命周期接口与状态冲突矩阵一致

### 阶段 8：MCP 适配层

#### 目标
让 AI agent / host 通过 MCP 调用 AIHub，而不复制内部逻辑。

#### 任务
- 实现 MCP server 入口
- 实现 `list_services`
- 实现 `get_service`
- 实现 `list_instances`
- 实现 `get_capabilities`
- 实现 `invoke_service`

#### 交付物
- MCP Adapter
- MCP 到 REST / 内部服务的映射层

#### 关键实现点
- MCP 只做协议外壳转换
- MCP 与 REST 必须复用相同的底层服务逻辑

#### 验收标准
- MCP 查询结果与 REST 查询结果一致
- MCP 调用结果与 REST 调用结果一致

### 阶段 9：子服务 SDK / 参考实现

#### 目标
降低子服务接入门槛，验证协议可实现性。

#### 任务
- 提供最小子服务 SDK
- 提供 manifest 模板
- 提供 capabilities 文档模板
- 提供 `service.register` / `service.heartbeat` / `invoke` 的参考实现
- 提供一个最小 demo 子服务

#### 交付物
- 子服务 SDK
- demo extension package
- 接入示例文档

#### 验收标准
- 新子服务可基于模板在低改动下接入
- demo 子服务可完成完整注册、心跳、调用和断连流程

### 阶段 10：测试、联调与收口

#### 目标
验证协议、状态机、生命周期和对外接口在真实流程中一致。

#### 任务
- 编写单元测试
- 编写集成测试
- 编写协议错误场景测试
- 编写生命周期状态冲突测试
- 编写断线、心跳超时、排空窗口测试
- 编写启动恢复测试
- 编写 REST 与 MCP 一致性测试

#### 交付物
- 自动化测试集
- 联调脚本
- 异常场景验证记录

#### 验收标准
- 关键协议路径均有自动化覆盖
- 所有状态冲突矩阵都被验证
- 更新、卸载、断线、超时等复杂流程都已联调通过

不建议跳过前置阶段直接实现调用链路，因为：

- 没有持久化服务定义就没有稳定服务身份
- 没有生命周期状态机就无法保证更新和卸载安全
- 没有连接与协议层就无法稳定支撑 REST 和 MCP

## 6. 每阶段最小完成定义

### 6.1 可进入下一阶段的标准
每个阶段至少满足以下条件才应进入下一阶段：

- 有明确的可运行交付物
- 有最小自动化验证
- 与前一阶段状态模型无冲突
- 不引入新的未决协议分支

### 6.2 不可跳过的验收点

- manifest 扫描与恢复必须先完成
- 生命周期状态冲突矩阵必须先落地
- WebSocket 注册和心跳必须先稳定
- `draining` 行为必须先打通
- REST 对外调用必须建立在内部调用闭环稳定之后

## 7. 建议任务包粒度

建议将工程任务拆成以下粒度：

- 一个任务包只负责一个核心模块或一个协议面
- 避免把生命周期、路由和 REST 混成一个大任务
- 每个任务包必须能独立验收

推荐任务包示例：

1. manifest 解析与启动恢复
2. 扩展包安装与启用/禁用
3. 更新源检查与手动更新
4. 卸载与 `draining` 联动
5. WebSocket Gateway
6. JSON-RPC 注册与心跳
7. Invocation 生命周期与调度
8. REST 查询与能力说明接口
9. REST 调用接口
10. MCP 适配层
11. 子服务 SDK 与 demo
12. 测试与联调

---

## 8. 风险与优先级

### 8.1 高优先级风险
- 生命周期状态机未先收敛，导致更新和卸载流程混乱
- 注册与心跳协议实现不一致，导致实例状态漂移
- `downstream_error` 与顶层协议错误混淆，导致调用方误判
- `draining` 未正确实现，导致进行中调用被强制切断

### 8.2 中优先级风险
- REST 与 MCP 映射不一致
- 能力说明文档和 manifest 脱节
- 调试接口保留策略实现不一致

### 8.3 优先级原则
优先保证：

1. 生命周期正确
2. 连接状态正确
3. 路由与调用正确
4. 对外接口完整
5. SDK 和生态体验

---

## 9. 总结

AIHub v1.0 的实现不应从 REST 接口开始，而应从“服务定义恢复、生命周期状态机、WebSocket 连接、内部控制协议、调用生命周期”这些底层核心先打牢。只有在这些基础稳定后，对外 REST / MCP、能力说明查询和子服务 SDK 才会自然收敛。

本文档的目标是把 AIHub 从设计阶段推进到工程执行阶段。后续实现应严格以阶段依赖和状态模型为准，避免跳步开发造成协议与行为分裂。
