## 1. 架构概述

### 1.1 架构定位
AIHub 是一个本地能力路由器与协议层，其核心职责是统一管理子服务的安装与运行状态，维护子服务与 AIHub 的长连接关系，接受调用方请求，将请求路由到合适的在线实例，并在运行时完成超时控制、异常清理和错误分层。

AIHub 在架构上只统一底层控制面与传输面，不统一上层业务协议。调用方与子服务之间的业务 payload 由双方自行约定，AIHub 在系统中只做透明透传。

### 1.2 架构目标
本架构需要同时满足以下要求：

- 对调用方提供统一的对外访问入口
- 对子服务提供统一的安装、启用、禁用、更新、卸载管理能力
- 对运行时实例提供统一的连接、注册、心跳和状态维护能力
- 对调用请求提供统一的路由、调度、转发与回传能力
- 对错误提供统一的分层表达
- 对上层 payload 保持协议中立

### 1.3 技术选型总览
AIHub v1.0 的核心技术选型如下：

- 对外入口：REST + MCP Adapter
- 对内连接：WebSocket
- 内部控制协议：JSON-RPC 风格消息模型
- 服务定义来源：manifest
- 安装单元：扩展包
- 运行时状态存储：内存注册表
- 请求关联标识：`invocationId` / request id
- 健康检测：Ping/Pong + `lastSeen`
- 默认实例调度策略：`least_inflight`

## 2. 总体架构

### 2.1 模块划分
AIHub 由以下核心模块组成：

- Northbound API Layer
- MCP Adapter
- Router Core
- Service Connection Gateway
- Service Registry / Instance Manager
- Extension Lifecycle Manager
- Invocation Lifecycle Manager
- Health & Heartbeat Manager
- Admin / Metadata API

### 2.2 架构关系图


### 2.3 控制关系
整体控制关系如下：

1. Northbound API Layer 负责接收外部调用和管理请求。
2. Router Core 负责将调用请求路由到目标服务和目标实例。
3. Service Registry / Instance Manager 负责为 Router Core 提供服务状态和实例状态。
4. Invocation Lifecycle Manager 负责管理调用请求的整个生命周期。
5. Service Connection Gateway 负责与子服务建立和维护 WebSocket 连接。
6. Health & Heartbeat Manager 负责驱动实例在线状态刷新和异常下线。
7. Extension Lifecycle Manager 负责服务安装包和 manifest 的生命周期管理。
8. Admin / Metadata API 负责对外暴露服务状态、生命周期状态和能力说明。
9. 本地扩展目录和 manifest 存储负责承载“已安装服务定义”这一持久事实。

## 3. 模块职责

### 3.1 Northbound API Layer

#### 3.1.1 负责内容
- 提供对外 REST 入口
- 接收调用请求
- 接收服务管理请求
- 接收状态查询请求
- 接收能力说明查询请求
- 将请求交给 Router Core 或 Admin / Metadata API

#### 3.1.2 不负责内容
- 不维护子服务连接
- 不进行实例调度
- 不解释业务 payload
- 不直接处理运行时超时与挂起请求

#### 3.1.3 技术方案
- 采用 REST 作为默认对外主入口
- 所有 northbound 请求统一进入同一 API 层
- 调用入口与管理入口共享统一的认证前置假设：本地单机可信环境

### 3.2 MCP Adapter

#### 3.2.1 负责内容
- 向 AI 生态暴露 MCP 适配入口
- 将 MCP 调用转换为 AIHub 内部标准调用请求
- 将 AIHub 的管理与能力说明结果转换为 MCP 可消费的结果

#### 3.2.2 不负责内容
- 不替代内部子服务总线协议
- 不直接维护子服务连接
- 不解释业务 payload

#### 3.2.3 技术方案
- MCP 在 v1.0 中作为对外适配层
- MCP 仅复用 AIHub 的调用、状态查询和能力说明能力
- MCP 不进入 AIHub 内部 WebSocket 总线

### 3.3 Router Core

#### 3.3.1 负责内容
- 根据 `serviceId` 执行服务级路由
- 根据可选 `instanceId` 执行实例直达
- 在未指定实例时选择合适实例
- 在请求进入运行时前执行可路由性判断

#### 3.3.2 不负责内容
- 不解释业务 payload
- 不负责 WebSocket 连接维护
- 不负责安装包和 manifest 管理

#### 3.3.3 技术方案
- 以 `serviceId` 作为主路由键
- 以 `instanceId` 作为可选直达键
- 默认实例选择策略为 `least_inflight`
- 只依赖 Service Registry / Instance Manager 暴露的状态信息做决策

### 3.4 Service Connection Gateway

#### 3.4.1 负责内容
- 维护 AIHub 与子服务之间的 WebSocket 长连接
- 接收子服务连接
- 承载内部 JSON-RPC 风格控制协议消息
- 将 Invocation Lifecycle Manager 的请求发送给目标实例
- 将子服务返回的响应或错误回送给 Invocation Lifecycle Manager

#### 3.4.2 不负责内容
- 不做安装管理
- 不做能力说明生成
- 不解释业务 payload

#### 3.4.3 技术方案
- 对内采用 WebSocket 长连接
- 内部消息采用 JSON-RPC 风格结构
- Gateway 负责 request/response/notification 的传输与关联

### 3.5 Service Registry / Instance Manager

#### 3.5.1 负责内容
- 维护已安装服务定义
- 维护已启用/已禁用状态
- 维护实例在线状态
- 维护实例运行时元数据
- 为路由和管理面提供统一状态视图

#### 3.5.2 不负责内容
- 不负责扩展包安装本身
- 不负责 WebSocket 连接细节
- 不负责调用请求超时控制

#### 3.5.3 技术方案
- v1.0 使用内存注册表维护运行时状态
- 已安装服务定义和启用状态来源于本地扩展目录中的 manifest 与安装记录
- 实例状态来源于运行时动态注册和心跳更新

### 3.6 Extension Lifecycle Manager

#### 3.6.1 负责内容
- 安装扩展包
- 校验安装包内容
- 读取和校验 manifest
- 启用服务
- 禁用服务
- 手动检查更新
- 手动更新服务
- 卸载服务

#### 3.6.2 不负责内容
- 不负责实例调度
- 不负责调用请求转发
- 不负责解析子服务业务能力语义

#### 3.6.3 技术方案
- 以扩展包作为安装单元
- 以 manifest 作为服务定义数据源
- 生命周期操作与 Service Registry / Instance Manager 联动
- 更新与卸载都必须感知在线实例状态
- AIHub 启动时由 Extension Lifecycle Manager 扫描本地扩展目录并恢复已安装服务定义与启用状态

### 3.7 Invocation Lifecycle Manager

#### 3.7.1 负责内容
- 为每个请求分配 `invocationId`
- 创建挂起请求记录
- 将请求提交到 Service Connection Gateway
- 维护请求超时计时器
- 处理响应回传
- 处理调用取消和异常清理
- 在服务更新、禁用、卸载时执行 invocation 排空与清理控制

#### 3.7.2 不负责内容
- 不负责服务安装和服务启停
- 不负责路由策略选择
- 不解释业务 payload

#### 3.7.3 技术方案
- 所有调用请求统一纳入 invocation 生命周期
- 通过 `invocationId` / request id 关联请求和响应
- 维护挂起请求表
- 超时后统一清理挂起状态并返回 router-level 错误
- 服务进入更新、禁用或卸载流程时，Invocation Lifecycle Manager 负责将服务标记为 `draining`
- `draining` 状态下拒绝新请求进入，已有挂起请求允许在超时窗口内完成；超过窗口后统一失败并清理

### 3.8 Health & Heartbeat Manager

#### 3.8.1 负责内容
- 驱动 Ping/Pong 或同等心跳机制
- 维护实例 `lastSeen`
- 判断实例是否在线
- 判断连接超时与心跳丢失
- 触发实例下线和状态刷新

#### 3.8.2 不负责内容
- 不负责业务调用路由
- 不负责安装管理

#### 3.8.3 技术方案
- 采用 Ping/Pong + `lastSeen` 的健康模型
- 心跳状态更新写入 Service Registry / Instance Manager
- 心跳异常触发实例摘除与挂起请求清理

### 3.9 Admin / Metadata API

#### 3.9.1 负责内容
- 对外暴露服务列表
- 对外暴露实例状态
- 对外暴露生命周期状态
- 对外暴露子服务能力说明

#### 3.9.2 不负责内容
- 不负责转发业务调用
- 不负责运行时连接维护

#### 3.9.3 技术方案
- 通过 REST 提供管理与说明查询接口
- 数据来源为 manifest、服务注册表和运行时状态
- 提供面向 AI 的结构化能力说明结果

## 4. 技术方案说明

### 4.1 对外：REST + MCP Adapter

#### 4.1.1 REST
REST 是 AIHub v1.0 的默认主入口，原因如下：

- 适合承载调用入口和管理入口
- 适合表达资源状态和生命周期状态
- 易于调试和测试
- 易于为工程团队和普通调用方集成
- 适合提供结构化的能力说明查询接口

REST 承担的主要职责包括：

- 调用子服务
- 查询服务列表
- 查询实例状态
- 查询生命周期状态
- 查询子服务能力说明
- 执行安装、启用、禁用、更新、卸载等管理操作

#### 4.1.2 MCP Adapter
MCP 作为面向 AI 生态的适配层存在，其目标是降低 AI agent 或 host 对 AIHub 的接入成本。

MCP 适配层仅复用 AIHub 现有能力，不改变 AIHub 内部的路由模型和子服务连接模型。

### 4.2 对内：WebSocket + JSON-RPC 风格控制协议

#### 4.2.1 WebSocket
WebSocket 负责提供：

- 持续连接
- 在线感知
- 双向消息传递
- 动态接入与动态断连

WebSocket 是 AIHub 与子服务之间的唯一运行时连接通道。

#### 4.2.2 JSON-RPC 风格控制协议
JSON-RPC 风格控制协议负责提供：

- request / response 关联机制
- notification 模型
- 标准错误结构
- 注册与心跳消息的统一承载方式

该协议只服务于控制面，不约束上层业务 payload 格式。

### 4.3 manifest
manifest 用于承载服务定义和安装管理元数据，至少需要支持以下信息类型：

- 服务标识
- 服务版本
- 协议兼容信息
- 安装管理所需元数据
- 对外能力说明数据来源

manifest 的关键定位是：

- 用于 AIHub 识别“这是什么服务”
- 用于 AIHub 管理“这个服务是否可安装、可启用、可更新”
- 用于 AIHub 对外提供能力说明接口的数据基础

manifest 不是运行时连接协议，也不是业务 payload 协议。

### 4.4 内存注册表
v1.0 运行时状态采用内存注册表维护，原因如下：

- 本地单机场景足够
- 实现复杂度低
- 适合快速形成清晰运行时模型

内存注册表至少保存以下状态：

- 在线实例列表
- 每实例并发状态
- 每实例心跳状态
- 挂起 invocation 状态

需要明确区分两类状态：

- 持久化管理事实：已安装扩展包、manifest、已启用/已禁用状态、能力说明元数据
- 运行时内存状态：在线实例、心跳状态、并发状态、排空状态、挂起 invocation

AIHub 启动时必须先扫描本地扩展目录和 manifest，恢复服务定义与启用状态，然后再等待运行时实例重新上线。

## 5. 数据流与控制流

### 5.1 安装包安装流程

```text
扩展包安装请求
  -> Extension Lifecycle Manager
  -> 校验安装包
  -> 读取并校验 manifest
  -> 将服务定义写入 Service Registry
  -> 服务进入“已安装”状态
```

该流程的关键控制点：

- 未通过校验的扩展包不能进入注册表
- 服务定义必须先于运行时实例存在

### 5.2 启用流程

```text
启用请求
  -> Extension Lifecycle Manager
  -> 更新服务状态为“已启用”
  -> Service Registry 刷新可路由状态
  -> Router Core 允许该服务参与路由
```

禁用流程与之相反。禁用后：

- Router Core 不再接受该服务的新请求
- 服务进入 `draining` 状态
- 已有挂起请求在排空窗口内继续执行
- 超过排空窗口仍未完成的请求按 invocation 生命周期清理并失败

### 5.3 子服务上线注册流程

```text
子服务启动
  -> 与 Service Connection Gateway 建立 WebSocket 连接
  -> 发送动态注册消息
  -> Health & Heartbeat Manager 开始跟踪心跳
  -> Service Registry 写入实例在线状态
  -> Router Core 可将新请求路由到该实例
```

### 5.4 调用请求处理流程

```text
调用方请求
  -> Northbound API Layer / MCP Adapter
  -> Router Core
  -> 查询 Service Registry
  -> 选择服务与实例
  -> Invocation Lifecycle Manager 创建 invocation
  -> Service Connection Gateway 转发请求
  -> 子服务处理并返回结果
  -> Invocation Lifecycle Manager 结束 invocation
  -> Northbound API Layer 回传结果
```

该流程中的关键控制点：

- 路由前必须先验证服务是否存在、是否启用、是否有在线实例
- payload 在整个过程中不做业务解析
- 响应必须按 `invocationId` / request id 精确关联

### 5.5 更新流程

```text
手动检查更新请求
  -> Extension Lifecycle Manager
  -> 获取可更新状态

手动更新请求
  -> Extension Lifecycle Manager
  -> 将目标服务停用
  -> 标记服务进入 draining
  -> 阻止新请求进入
  -> 等待已有挂起请求排空或超时
  -> 断开或摘除旧实例
  -> 替换安装内容与 manifest
  -> 刷新服务定义
  -> 重新启用服务
  -> 等待新实例上线注册
```

该流程的关键控制点：

- 更新不是单纯替换文件，还必须联动路由状态和在线实例状态
- 更新前必须先进入排空状态，不能直接无条件切断所有进行中的调用
- 更新完成前，旧实例不能继续以失效状态参与路由

### 5.6 卸载流程

```text
卸载请求
  -> Extension Lifecycle Manager
  -> 停用目标服务
  -> 标记服务进入 draining
  -> 阻止新请求进入
  -> 等待已有挂起请求排空或超时
  -> 断开或摘除在线实例
  -> 清理 Service Registry 中的服务定义和状态
  -> 删除安装内容
```

该流程的关键控制点：

- 卸载前必须先停用
- 卸载前必须先进入排空状态，不能在存在进行中调用时直接切断服务
- 卸载时必须清理注册状态和在线状态
- 卸载后目标服务不可再被路由

### 5.7 超时与断线流程

```text
调用中超时 / 连接断线
  -> Invocation Lifecycle Manager 检测异常
  -> 清理挂起 invocation
  -> Service Registry 更新实例状态
  -> Router Core 不再选择该实例
  -> Northbound API Layer 返回 router-level 错误
```

## 6. 路由与调度

### 6.1 路由键
AIHub v1.0 的主路由键为 `serviceId`。调用方必须明确请求目标服务。

调用方可以额外指定 `instanceId`，用于直达某个实例。

### 6.2 服务级路由
Router Core 在处理请求时至少需要完成以下检查：

- 目标服务是否存在
- 目标服务是否已安装
- 目标服务是否已启用
- 目标服务是否存在可用在线实例

### 6.3 实例选择策略
当调用方未指定 `instanceId` 时，Router Core 采用 `least_inflight` 作为默认实例选择策略。

选择原因：

- 能够在多实例场景下降低热点实例堆积
- 比固定首实例或最早连接实例更适合长期运行

### 6.4 并发与队列
每个实例需要支持以下运行时控制参数：

- 并发上限
- 队列上限

Router Core 与 Invocation Lifecycle Manager 需要根据这些状态共同决定：

- 请求立即下发
- 请求进入等待
- 请求直接失败

## 7. 错误模型

### 7.1 错误分层
AIHub v1.0 的错误分为三层：

1. router-level error
2. transport-level error
3. downstream opaque error

### 7.2 router-level error
由 AIHub 路由层和控制层产生，典型场景包括：

- 服务不存在
- 服务已禁用
- 无在线实例
- 实例不可路由
- 调用超时
- 请求队列已满

这类错误需要由 AIHub 明确返回给调用方。

### 7.3 transport-level error
由连接层产生，典型场景包括：

- WebSocket 断开
- 注册失败
- 心跳丢失
- 协议消息非法

这类错误会影响实例状态，并可能进一步触发 router-level 错误。

### 7.4 downstream opaque error
由子服务返回的业务错误，AIHub 不解释其语义，只负责按调用关联原样回传。

AIHub 只需要保证：

- 错误来自哪一次 invocation 可被识别
- 调用方能区分其来源于下游子服务

---

## 8. 能力说明查询架构

### 8.1 能力说明的定位
AIHub 对外需要提供一个能力说明查询能力，用于返回指定子服务的：

- 能力列表
- 能力用途说明
- 调用方式
- 参数说明
- 返回格式说明

这项能力主要面向其他 AI、agent 或自动化系统，而不是单纯面向人工阅读。

### 8.2 数据来源
能力说明数据由以下来源组成：

- manifest 中的服务定义信息
- 子服务随安装包提供的能力说明元数据

### 8.3 架构要求
能力说明查询能力需要满足以下要求：

- 不依赖子服务当前是否在线
- 不要求 AIHub 理解能力的业务语义
- 只负责将结构化说明对外暴露
- 可通过 REST 和 MCP 两种对外入口访问

---

## 9. 非约束项

AIHub v1.0 在技术架构上明确以下事项不是架构约束：

- 子服务的具体运行形态
- 子服务的实现语言
- 子服务的业务协议类型
- 子服务的业务数据结构

只要子服务满足以下前提，即可纳入 AIHub 体系：

- 能以扩展包方式被安装
- 能提供合法 manifest
- 能在运行时通过 WebSocket 接入
- 能遵守 AIHub 的内部控制协议

---

## 10. 总结

AIHub v1.0 的技术架构以“控制面统一、运行时透明透传、安装与连接分离、服务定义与实例状态分离”为核心原则展开。整个系统通过 REST + MCP 对外暴露统一入口，通过 WebSocket + JSON-RPC 风格控制协议与子服务通信，通过路由、注册、生命周期和健康管理等模块协同完成本地子服务生态的统一底层支撑。
