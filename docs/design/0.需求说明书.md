- **负责人**: 项目负责人

## 1. 需求背景

### 1.1 现状分析
当前 AIHub 系统已实现:
- WebSocket 服务器 + REST API 服务
- 通过 WebSocket 向浏览器扩展(aiClaw/TweetClaw)下发命令
- 扩展能够正常工作并执行任务

### 1.2 存在的问题
- 代码和逻辑硬编码,缺乏灵活性
- 新增软件接入需要修改核心代码
- 协议不开放,无法被第三方软件复用
- AI 调用第三方软件的成本高

### 1.3 需求动机
降低 AI 调用第三方软件的接入成本,通过标准化协议让任意软件成为可被调用的服务提供者。

---

## 2. 核心目标

### 2.1 主要目标
将 AIHub 从"硬编码的命令分发器"升级为"能力注册与调度中心",实现:
1. **标准化协议**: 定义开放的通信协议和能力声明规范
2. **低成本接入**: 任何软件通过配置文件即可接入
3. **统一调用**: AI 和其他服务通过标准 REST API 调用所有接入的软件能力
4. **可扩展性**: 为未来功能演进预留设计空间

### 2.2 非目标(本期不做)
- 安全认证和权限管理(默认本地可信环境)
- 异步任务支持
- 流式响应支持
- 事件推送机制
- x402 付费协议

## 3. 系统架构概览

### 3.1 核心组件
```
┌─────────────────┐
│  REST API 调用者 │ (AI/脚本/其他服务)
└────────┬────────┘
         │ HTTP REST API
         ↓
┌─────────────────────────────┐
│      AIHub LocalBridge      │
│  ┌─────────────────────┐   │
│  │  能力注册表         │   │
│  │  (从配置文件加载)   │   │
│  └─────────────────────┘   │
│  ┌─────────────────────┐   │
│  │  REST API 路由      │   │
│  │  /api/{软件名}/{能力}│   │
│  └─────────────────────┘   │
│  ┌─────────────────────┐   │
│  │  WebSocket 服务器   │   │
│  └─────────────────────┘   │
└──────────┬──────────────────┘
           │ WebSocket
           ↓
┌──────────────────────────────┐
│  软件客户端 (aiClaw/其他)    │
│  - 实现 WebSocket 客户端     │
│  - 提供配置文件声明能力      │
└──────────────────────────────┘
```

### 3.2 工作流程
1. **启动阶段**: AIHub 加载所有软件的配置文件,构建能力注册表
2. **连接阶段**: 软件启动后通过 WebSocket 连接到 AIHub
3. **调用阶段**: REST API 调用者发起请求 → AIHub 路由 → WebSocket 转发 → 软件执行 → 返回结果

## 4. 核心场景

### 4.1 场景一: 软件接入
**角色**: 第三方软件开发者

**前置条件**:
- AIHub LocalBridge 已启动
- 开发者已实现 WebSocket 客户端

**操作步骤**:
1. 开发者编写配置文件,声明软件提供的能力
2. 将配置文件放置到 AIHub 指定目录
3. 启动软件,连接到 AIHub WebSocket 服务器
4. AIHub 验证连接并激活该软件的能力

**预期结果**:
- 软件成功连接到 AIHub
- 软件声明的能力可通过 REST API 被调用

**技术设计点**:
> **[架构师设计任务 #1: 配置文件格式设计]**
>
> 请设计软件能力声明的配置文件格式,需要满足以下要求:
> 1. **基本信息**: 软件标识(名称/ID/版本)、描述、作者等元数据
> 2. **能力声明**: 每个能力需要包含:
>    - 能力名称(用于 REST API 路径)
>    - 能力描述(人类可读)
>    - 输入参数定义(参数名、类型、是否必填、默认值、描述)
>    - 输出结果定义(返回值类型、结构描述)
>    - 示例(输入输出示例)
> 3. **连接信息**: WebSocket 连接相关配置
> 4. **扩展性**: 为未来版本(异步任务/流式响应/事件推送)预留字段
> 5. **灵活性**: 支持复杂数据类型(对象/数组/嵌套结构)
> 6. **规范性**: 避免混乱,提供清晰的验证规则
>
> 请提供:
> - 配置文件格式规范(JSON/YAML/TOML 等)
> - 完整的 Schema 定义
> - 至少 2 个实际示例(aiClaw 和 TweetClaw)
> - 配置文件的加载和验证机制设计

### 4.2 场景二: REST API 调用
**角色**: AI 或其他服务

**前置条件**:
- 目标软件已接入 AIHub
- 目标软件的 WebSocket 连接正常

**操作步骤**:
1. 调用者发起 HTTP 请求到 `/api/{软件名}/{能力名}`
2. 请求 body 包含能力所需的参数
3. AIHub 接收请求,查找能力注册表
4. AIHub 通过 WebSocket 将请求转发给目标软件
5. 目标软件执行能力,返回结果
6. AIHub 将结果返回给调用者

**预期结果**:
- 调用者收到正确的响应结果
- 响应时间在可接受范围内(同步调用)

**技术设计点**:
> **[架构师设计任务 #2: WebSocket 通信协议设计]**
>
> 请设计 AIHub 与软件客户端之间的 WebSocket 通信协议,需要满足以下要求:
> 1. **消息格式**: 定义标准的消息结构(请求/响应/错误)
> 2. **请求路由**: 如何在 WebSocket 消息中标识目标能力
> 3. **请求追踪**: 如何关联 REST API 请求和 WebSocket 响应(请求 ID 机制)
> 4. **超时处理**: 同步调用的超时时间和超时响应机制
> 5. **并发处理**: 如何处理同一软件的多个并发请求
> 6. **连接管理**: 心跳机制、重连策略、连接状态管理
> 7. **扩展性**: 为异步任务、流式响应、事件推送预留协议字段
>
> 请提供:
> - 完整的消息格式定义(JSON Schema)
> - 请求-响应流程的序列图
> - 错误处理流程
> - 协议版本管理机制

### 4.3 场景三: 错误处理
**角色**: REST API 调用者

**异常情况**:
1. 目标软件未连接
2. 目标软件连接超时
3. 目标软件执行失败
4. 参数验证失败
5. 能力不存在

**预期结果**:
- 返回标准化的错误响应
- 错误码清晰表达错误类型
- 错误信息包含足够的调试信息

**技术设计点**:
> **[架构师设计任务 #3: 标准错误码体系设计]**
>
> 请设计 AIHub 协议的标准错误码体系,需要满足以下要求:
> 1. **分类清晰**: 类似 HTTP 状态码,按错误类型分类(4xx 客户端错误/5xx 服务端错误)
> 2. **覆盖全面**: 涵盖所有可能的异常场景:
>    - 连接相关错误(软件离线/连接超时/连接中断)
>    - 请求相关错误(参数错误/能力不存在/权限不足)
>    - 执行相关错误(执行超时/执行失败/资源不足)
>    - 协议相关错误(协议版本不匹配/消息格式错误)
> 3. **可扩展**: 为未来新增错误类型预留编码空间
> 4. **标准化**: 错误响应的统一格式(错误码/错误消息/详细信息/追踪 ID)
> 5. **国际化**: 支持多语言错误消息
>
> 请提供:
> - 完整的错误码列表和说明
> - 错误响应格式定义
> - 错误处理最佳实践指南
> - 与 HTTP 状态码的映射关系

### 4.4 场景四: 能力发现
**角色**: AI 或开发者

**需求**:
- 查询 AIHub 当前接入了哪些软件
- 查询某个软件提供了哪些能力
- 查询某个能力的详细信息(参数/返回值)

**预期结果**:
- 提供 REST API 端点查询能力注册表
- 返回结构化的能力信息

**技术设计点**:
> **[架构师设计任务 #4: 能力发现 API 设计]**
>
> 请设计能力发现相关的 REST API,需要满足以下要求:
> 1. **列出所有软件**: `GET /api/services` 返回所有接入的软件列表
> 2. **查询软件详情**: `GET /api/services/{软件名}` 返回软件信息和能力列表
> 3. **查询能力详情**: `GET /api/services/{软件名}/{能力名}` 返回能力的完整定义
> 4. **健康检查**: `GET /api/services/{软件名}/health` 返回软件连接状态
> 5. **过滤和搜索**: 支持按标签/分类/关键词搜索能力
>
> 请提供:
> - 完整的 API 端点定义(OpenAPI/Swagger 格式)
> - 响应数据结构
> - 分页和过滤机制

## 5. 技术约束

### 5.1 网络约束
- AIHub LocalBridge 默认监听 `127.0.0.1`,仅本地访问
- WebSocket 和 REST API 共享同一端口
- 不考虑跨域和远程访问场景

### 5.2 性能约束
- 同步调用响应时间 < 30 秒(可配置)
- 支持至少 10 个软件同时接入
- 支持至少 100 个并发 REST API 请求

### 5.3 兼容性约束
- 配置文件格式需要人类可读可编辑
- WebSocket 协议需要主流语言都有成熟的客户端库
- REST API 遵循 RESTful 规范

## 10. 附录

### 10.1 术语表
- **AIHub LocalBridge**: 本地运行的 AIHub 服务,提供 WebSocket 和 REST API
- **能力(Capability)**: 软件提供的可被调用的功能单元
- **能力注册表**: AIHub 维护的所有软件能力的索引
- **同步调用**: 请求-响应模式,调用者等待结果返回
- **异步任务**: 提交任务后立即返回任务 ID,后续查询结果
- **流式响应**: 结果分块逐步返回,适合大数据或实时场景
- **事件推送**: 软件主动向 AIHub 推送事件,订阅者接收

### 10.2 参考资料
- WebSocket 协议: RFC 6455
- RESTful API 设计规范: REST API Design Rulebook
- OpenAPI 规范: https://swagger.io/specification/
- JSON Schema: https://json-schema.org/
