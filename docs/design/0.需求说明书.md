## 1. 需求背景

### 1.1 现状分析
当前 AIHub 系统已实现:
- WebSocket 服务器 + REST API 服务
- 通过 WebSocket 向浏览器扩展(aiClaw/TweetClaw)下发命令
- 扩展能够正常工作并执行任务

### 1.2 存在的问题
- 代码和逻辑硬编码,缺乏灵活性
- 新增软件接入需要修改核心代码
- 协议不开放,无法被第三方软件复用
- AI 调用第三方软件的成本高

### 1.3 需求动机
降低 AI 调用第三方软件的接入成本,通过标准化协议让任意软件成为可被调用的服务提供者。

---

## 2. 核心目标

### 2.1 主要目标
将 AIHub 从"硬编码的命令分发器"升级为"能力注册与调度中心",实现:
1. **标准化协议**: 定义开放的通信协议和能力声明规范
2. **低成本接入**: 任何软件通过配置文件即可接入
3. **统一调用**: AI 和其他服务通过标准 REST API 调用所有接入的软件能力
4. **可扩展性**: 为未来功能演进预留设计空间

### 2.2 非目标(本期不做)
- 安全认证和权限管理(默认本地可信环境)
- 异步任务支持
- 流式响应支持
- 事件推送机制
- x402 付费协议

## 3. 系统架构概览

### 3.1 核心组件
```
┌─────────────────┐
│  GraphQL 调用者  │ (AI/脚本/其他服务)
└────────┬────────┘
         │ GraphQL over HTTP
         ↓
┌─────────────────────────────────────┐
│      AIHub LocalBridge              │
│      (GraphQL Gateway)              │
│  ┌─────────────────────────────┐   │
│  │  Schema → Service 映射表    │   │
│  │  (从配置文件加载, O(1)查找) │   │
│  └─────────────────────────────┘   │
│  ┌─────────────────────────────┐   │
│  │  GraphQL 路由引擎           │   │
│  │  1. 提取 schema             │   │
│  │  2. 提取 operation          │   │
│  │  3. 透传请求和响应          │   │
│  └─────────────────────────────┘   │
│  ┌─────────────────────────────┐   │
│  │  WebSocket 服务器           │   │
│  │  (graphql-ws 协议)          │   │
│  └─────────────────────────────┘   │
└──────────┬──────────────────────────┘
           │ WebSocket (graphql-ws)
           ↓
┌──────────────────────────────────────┐
│  子服务 (aiClaw/TweetClaw/其他)     │
│  - 实现 GraphQL Resolver            │
│  - 理解并执行 GraphQL 操作          │
│  - 提供配置文件声明能力             │
│  - 返回标准 GraphQL 响应            │
└──────────────────────────────────────┘
```

### 3.2 工作流程
1. **启动阶段**: AIHub 加载所有子服务的配置文件,构建 Schema → Service 映射表
2. **连接阶段**: 子服务启动后通过 WebSocket (graphql-ws 协议) 连接到 AIHub
3. **调用阶段**:
   - GraphQL 调用者发起请求(携带 schema 和 operation)
   - AIHub 两步路由: schema → service, operation → capability
   - 通过 WebSocket 转发完整 GraphQL 请求
   - 子服务执行并返回标准 GraphQL 响应
   - AIHub 透传响应给调用者

## 4. 核心场景

### 4.1 场景一: 软件接入
**角色**: 第三方软件开发者

**前置条件**:
- AIHub LocalBridge 已启动
- 开发者已实现 WebSocket 客户端

**操作步骤**:
1. 开发者编写配置文件,声明软件提供的能力
2. 将配置文件放置到 AIHub 指定目录
3. 启动软件,连接到 AIHub WebSocket 服务器
4. AIHub 验证连接并激活该软件的能力

**预期结果**:
- 软件成功连接到 AIHub
- 软件声明的能力可通过 REST API 被调用

**技术设计点**:
> **[架构师设计任务 #1: 配置文件格式设计]** ✅ 已完成
>
> **设计决策**:
> - 采用 GraphQL + graphql-ws 协议标准
> - AIHub 作为 GraphQL Gateway,完全透传数据
> - 配置文件仅用于路由映射(schema → service, operation → capability)
> - 两步路由机制,O(1) 查找性能
>
> **配置文件格式**:
> ```json
> {
>   "version": "1.0",
>   "service": {
>     "name": "aiClaw",
>     "schema": "com.aihub.aiclaw",
>     "description": "AI platform automation service"
>   },
>   "capabilities": [
>     {
>       "operation": "queryAIStatus",
>       "type": "query",
>       "description": "Query current AI platform tabs status"
>     },
>     {
>       "operation": "sendAIMessage",
>       "type": "mutation",
>       "description": "Send message to AI platform"
>     }
>   ]
> }
> ```
>
> 详细设计见: [关键技术验证说明书 - 验证项 #1](./1.关键技术验证说明书.md#验证项-1-配置文件格式标准)

### 4.2 场景二: GraphQL API 调用
**角色**: AI 或其他服务

**前置条件**:
- 目标子服务已接入 AIHub
- 目标子服务的 WebSocket 连接正常

**操作步骤**:
1. 调用者发起 GraphQL 请求(HTTP POST),携带 schema 和 operation
   ```json
   {
     "query": "query { queryAIStatus { tabs { platform tabId url } } }",
     "variables": {},
     "extensions": {
       "schema": "com.aihub.aiclaw"
     }
   }
   ```
2. AIHub 接收请求,执行两步路由:
   - 第一步: 根据 `extensions.schema` 定位目标服务(O(1) 查找)
   - 第二步: 根据 operation name 验证服务是否支持此能力
3. AIHub 通过 WebSocket (graphql-ws 协议) 转发完整 GraphQL 请求
4. 子服务执行 GraphQL 操作,返回标准 GraphQL 响应
5. AIHub 透传响应给调用者

**预期结果**:
- 调用者收到标准 GraphQL 响应 `{data: {...}, errors: [...]}`
- 响应时间在可接受范围内(同步调用)

**技术设计点**:
> **[架构师设计任务 #2: WebSocket 通信协议设计]**
>
> 请基于 graphql-ws 协议标准设计 AIHub 与子服务之间的通信机制,需要满足以下要求:
> 1. **协议标准**: 采用 graphql-ws 协议(connection_init, subscribe, next, error, complete)
> 2. **消息格式**: 标准 GraphQL 请求和响应格式
> 3. **请求追踪**: graphql-ws 的 ID 机制关联请求和响应
> 4. **超时处理**: 同步调用的超时时间和超时响应机制
> 5. **并发处理**: 如何处理同一子服务的多个并发请求
> 6. **连接管理**: 心跳机制、重连策略、连接状态管理
> 7. **透传机制**: AIHub 如何完全透传 GraphQL 请求和响应
>
> 请提供:
> - graphql-ws 协议的具体实现细节
> - 请求-响应流程的序列图
> - 错误处理流程
> - 连接管理机制

### 4.3 场景三: 错误处理
**角色**: REST API 调用者

**异常情况**:
1. 目标软件未连接
2. 目标软件连接超时
3. 目标软件执行失败
4. 参数验证失败
5. 能力不存在

**预期结果**:
- 返回标准化的错误响应
- 错误码清晰表达错误类型
- 错误信息包含足够的调试信息

**技术设计点**:
> **[架构师设计任务 #3: 标准错误码体系设计]**
>
> 请基于 GraphQL 错误规范设计 AIHub 的错误处理体系,需要满足以下要求:
> 1. **GraphQL 标准**: 采用标准 GraphQL 错误格式 `{errors: [{message, extensions: {code}}]}`
> 2. **错误分类**: 定义清晰的错误码:
>    - 路由相关错误(SCHEMA_NOT_FOUND, OPERATION_NOT_SUPPORTED)
>    - 连接相关错误(SERVICE_OFFLINE, CONNECTION_TIMEOUT)
>    - 执行相关错误(EXECUTION_TIMEOUT, EXECUTION_FAILED)
>    - 协议相关错误(INVALID_GRAPHQL_REQUEST, PROTOCOL_ERROR)
> 3. **透传机制**: 子服务返回的 GraphQL 错误如何透传
> 4. **可扩展**: 为未来新增错误类型预留编码空间
> 5. **国际化**: 支持多语言错误消息
>
> 请提供:
> - 完整的错误码列表和说明
> - GraphQL 错误响应格式定义
> - 错误处理最佳实践指南
> - 与 HTTP 状态码的映射关系

## 5. 技术约束

### 5.1 网络约束
- AIHub LocalBridge 默认监听 `127.0.0.1`,仅本地访问
- WebSocket 和 REST API 共享同一端口
- 不考虑跨域和远程访问场景

### 5.2 性能约束
- 同步调用响应时间 < 30 秒(可配置)
- 支持至少 10 个软件同时接入
- 支持至少 100 个并发 REST API 请求

### 5.3 兼容性约束
- 配置文件格式需要人类可读可编辑
- WebSocket 协议需要主流语言都有成熟的客户端库
- REST API 遵循 RESTful 规范

## 10. 附录

### 10.1 术语表
- **AIHub LocalBridge**: 本地运行的 AIHub 服务,提供 WebSocket 和 REST API
- **能力(Capability)**: 软件提供的可被调用的功能单元
- **能力注册表**: AIHub 维护的所有软件能力的索引
- **同步调用**: 请求-响应模式,调用者等待结果返回
- **异步任务**: 提交任务后立即返回任务 ID,后续查询结果
- **流式响应**: 结果分块逐步返回,适合大数据或实时场景
- **事件推送**: 软件主动向 AIHub 推送事件,订阅者接收

### 10.2 参考资料
- WebSocket 协议: RFC 6455
- RESTful API 设计规范: REST API Design Rulebook
- OpenAPI 规范: https://swagger.io/specification/
- JSON Schema: https://json-schema.org/
