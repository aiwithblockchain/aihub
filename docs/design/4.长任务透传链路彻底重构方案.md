## 1. 文档目标

本文档用于一次性彻底解决当前 AIHub / LocalBridge 在长任务透传场景下暴露出的结构性问题。

本文档覆盖以下范围：

- 当前长任务链路的真实问题归因
- 目标架构与目标协议
- 各模块的新职责边界
- 长任务生命周期与状态机
- Go WebSocket 层的彻底重构要求
- REST / WebSocket / 扩展内部协议的变更方案
- 代码落地范围与实施顺序
- 最终验收标准

本文档不提供回退方案，不提供兼容旧设计的兜底路径。  
目标是用单一新架构替换当前同步桥接式长任务实现。

## 2. 问题定义与范围

### 2.1 核心边界

本次重构首先要明确系统边界。

`LocalBridgeMac / Go bridge / Xcode 工程` 的职责不是理解某个具体业务，而是：

- 接收外部 REST 请求
- 选择目标插件实例
- 透传控制请求
- 维护调用生命周期
- 维护连接、心跳、状态与错误语义

它不应：

- 承担具体业务执行
- 绑定某一种业务场景的专用时序
- 因某个业务长事务而改变为同步阻塞式桥接器

因此，本次要解决的问题不是“视频上传怎么做”，而是：

```text
AIHub 如何正确承载任意长任务的发起、执行、进度回传、失败表达和结果获取
```

视频上传只是第一个暴露该问题的高压力场景。

### 2.2 当前典型长任务链路

以视频上传为例，当前长任务链路实际经过如下路径：

```text
Python CLI
  -> REST /api/v1/x/media/upload
  -> Go bridge
  -> tweetClaw background
  -> content script
  -> page injection / X upload API
  -> 返回同一条 WebSocket 回调
  -> REST 同步返回
```

### 2.3 已确认的真实问题

当前问题不是单点 bug，而是长任务承载模型错误导致的三个结构性问题叠加。

#### 问题 A：长任务被错误建模为同步 request/response

当前桥接模型默认假设：

- 外部一个 REST 请求
- 内部一个 WebSocket request
- 插件最终回一个 response
- HTTP 在同一生命周期内同步结束

这个模型适合短任务，不适合长任务。

一旦任务执行时间较长，就会引出以下问题：

- HTTP 调用长时间挂起
- WebSocket 连接生命周期和任务生命周期被错误绑定
- 插件重连、心跳抖动、service worker 生命周期变化会直接杀死任务
- 调用方无法获得结构化进度，只能等待最终结果或超时

#### 问题 B：控制面和数据面没有分离

视频上传只是把这一问题暴露得最明显。

当前系统将以下内容混在一起：

- 控制命令
- 状态同步
- 大 payload 数据传输
- 最终结果回传

这导致：

- 大 payload 会冲击控制链路
- 数据面异常会影响控制面稳定性
- 控制层无法保持简单、稳定、可观察

视频场景中的 `tabs.sendMessage` `64MiB` 上限，本质上是数据面错误占用了控制面通道。

#### 问题 C：Go WebSocket 写模型不安全

当前 Go 服务存在多个写入口：

- 业务响应
- `hello_ack`
- `pong`
- 后续还可能出现长任务事件

这些写入共享同一个 gorilla/websocket `Conn`，已实锤导致：

```text
panic: concurrent write to websocket connection
```

这说明 LocalBridge 当前并不是一个正确的长连接运行时网关。

### 2.4 当前方案为什么本质上不稳

即使暂时修好某个业务路径，当前架构仍然具备以下不可接受特征：

- 长任务不是独立任务对象，而是被绑死在同步调用上
- 控制面和数据面混用
- Go bridge 不是严格的单写出站模型
- 失败面太多，且失败后缺少稳定状态恢复语义
- 调用方无法以统一方式获取进度、状态和错误

结论：

当前设计不是“某个接口需要修补”，而是“长任务透传架构需要整体替换”。

## 3. 重构目标

### 3.1 总体目标

建立一个满足以下要求的新长任务透传架构：

- 适用于任意长任务，而不局限于视频上传
- 不让长任务占用同步桥接调用生命周期
- 控制面与数据面强分离
- 不让 WebSocket 心跳与业务写并发冲突
- 长任务具备明确的状态、进度和错误模型
- 调用方通过任务接口查询结果，而不是盲等一个 HTTP 请求

### 3.2 强制目标

新架构必须满足以下硬性约束：

- WebSocket 层所有写操作必须通过单一串行发送模型完成
- 所有长任务必须改为任务模型，不再使用“单个 HTTP 请求等待整个执行完成”
- 数据面不得通过控制面传输
- 长任务状态必须可查询
- 错误必须分层表达，且直接返回给调用方

### 3.3 明确废弃的旧设计

以下设计在重构后必须被彻底移除：

- 任意“同步桥接长任务”设计
- 任意“大 payload 走 WebSocket 或扩展控制消息”设计
- 任意“业务执行结果必须通过单次 response 才能完成调用”的长任务模型
- Go 服务任意路径直接对同一 `Conn` 调用 `WriteMessage`

## 4. 新架构总览

### 4.1 新架构原则

新架构采用以下五条原则：

1. 控制面与数据面分离
2. 长事务任务化
3. 任务状态化
4. WebSocket 发送单线程化
5. 平台能力通用化

### 4.2 新架构关系图

```text
Python CLI
  -> REST create long task
  -> REST push task input data (optional)
  -> REST start task
  -> REST poll task status
  -> REST fetch task result

Go REST Layer
  -> Long Task Manager
  -> Task Data Store
  -> WebSocket Control Gateway

tweetClaw background
  -> receives start/cancel/query task commands
  -> loads task input from local data store
  -> coordinates plugin-local executor

plugin business executor
  -> executes domain-specific long task
  -> reports progress / phase / result back to background

Go WebSocket Server
  -> one writer loop per session
  -> request/response and event notifications share same send queue
```

### 4.3 新的基本判断

在新架构下：

- 长任务是独立任务对象，不是同步调用
- WebSocket 只负责控制命令和状态事件
- 大输入数据由任务数据面承载，不由控制面承载
- CLI 通过任务状态接口感知进度和最终结果

## 5. 新模块职责边界

### 5.1 LocalBridgeMac / Xcode 工程

Xcode 工程的职责严格收敛为平台网关：

- 提供 northbound REST API
- 管理任务元数据与状态
- 维护插件会话
- 将控制命令路由到目标插件
- 接收插件状态事件和结果

Xcode 工程不负责：

- 执行具体业务
- 理解视频上传细节
- 为某个业务定制同步长事务时序

### 5.2 Python CLI

CLI 只负责：

- 创建长任务
- 向任务数据面上传输入数据
- 触发任务开始执行
- 轮询任务状态
- 获取任务结果
- 执行后续业务动作

CLI 不再负责：

- 长时间阻塞等待一个同步桥接响应

### 5.3 Go REST Layer

Go REST 层负责：

- 创建长任务记录
- 接收并保存任务输入数据
- 提供任务输入数据读取接口给插件
- 启动任务
- 查询任务状态
- 接收扩展上报的阶段、进度、错误与最终结果

Go REST 层不负责：

- 解释具体业务细节
- 执行具体业务任务

### 5.4 Long Task Manager

新增独立模块：`Long Task Manager`

负责：

- 维护长任务生命周期
- 存储任务元数据
- 维护状态机
- 记录失败原因和阶段
- 提供任务查询接口

最小任务字段：

- `taskId`
- `taskKind`
- `clientName`
- `instanceId`
- `ownerClientName`
- `ownerInstanceId`
- `inputMode`
- `state`
- `phase`
- `progress`
- `inputRef`
- `errorCode`
- `errorMessage`
- `resultRef`
- `createdAt`
- `updatedAt`

### 5.5 Task Data Store

新增独立模块：`Task Data Store`

负责：

- 接收调用方传入的大输入
- 持久化到本地临时目录
- 为插件提供按 `taskId` 读取输入数据的能力
- 提供校验信息

约束：

- task data store 是数据面
- WebSocket 不承担文件数据传输

### 5.6 WebSocket Control Gateway

WebSocket 只负责控制面：

- `request.start_task`
- `request.cancel_task`
- `event.task_progress`
- `event.task_failed`
- `event.task_completed`

WebSocket 不负责：

- 大文件内容传输
- 单个长任务的全部结果同步回传

### 5.7 插件 background

background 负责：

- 接收任务开始命令
- 建立插件内任务执行上下文
- 从 Go task data store 拉取输入数据
- 调度插件内业务执行器
- 上报任务进度和阶段

background 不再负责：

- 承担大对象控制消息传输
- 依赖单次 response 表达长任务最终结果

### 5.8 插件业务执行器

业务执行器负责：

- 执行具体业务长任务
- 上报阶段和进度
- 输出具体业务错误
- 产出业务结果

业务执行器不负责：

- 管理任务生命周期全局状态
- 管理 northbound REST 语义
- 管理平台级连接和路由

### 5.9 Go WebSocket Session Writer

新增明确要求：

- 每个 session 必须有唯一 writer loop
- 所有写操作必须统一进入 send queue
- 任何路径不得直接调用 `Conn.WriteMessage`

这是新架构的硬约束，不是建议项。

## 6. 新长任务模型

### 6.1 状态机

长任务状态机定义如下：

```text
created
  -> receiving_input
  -> ready
  -> starting
  -> running
  -> completed

任意中间态
  -> failed
  -> cancelled
```

### 6.2 状态说明

- `created`：任务已创建，尚未接收输入数据
- `receiving_input`：调用方正在向 task data store 写入输入数据
- `ready`：所有输入已完成接收，可以启动执行
- `starting`：Go 已向扩展下发开始执行命令
- `running`：插件正在执行任务
- `completed`：任务成功，结果已可读
- `failed`：任务失败
- `cancelled`：任务被调用方取消

### 6.3 阶段字段

除 `state` 外，还必须单独保留 `phase`：

- 由具体任务类型定义
- `done`

这样调用方能够区分：

- 任务生命周期状态
- 具体业务执行阶段

### 6.4 进度语义

任务必须至少上报：

- `progress`
- `currentPhase`
- `startedAt`
- `updatedAt`

进度更新粒度：

- 至少每个业务阶段变化上报一次
- 对长时间阶段必须周期性上报

## 7. 新协议设计

### 7.1 REST API 设计

#### 7.1.1 创建长任务

`POST /api/v1/tasks`

请求：

```json
{
  "clientName": "tweetClaw",
  "taskKind": "x.media_upload",
  "inputMode": "chunked_binary",
  "tabId": 305047737
}
```

响应：

```json
{
  "taskId": "task_xxx",
  "state": "created",
  "inputMode": "chunked_binary"
}
```

#### 7.1.2 写入任务输入

`PUT /api/v1/tasks/{taskId}/input/{partIndex}`

请求体：

- 原始二进制 part

请求头：

- `Content-Type: application/octet-stream`

响应：

```json
{
  "ok": true,
  "receivedParts": 12
}
```

#### 7.1.3 完成输入写入

`POST /api/v1/tasks/{taskId}/seal`

响应：

```json
{
  "ok": true,
  "state": "ready"
}
```

#### 7.1.4 启动任务

`POST /api/v1/tasks/{taskId}/start`

响应：

```json
{
  "ok": true,
  "state": "starting"
}
```

#### 7.1.5 查询任务

`GET /api/v1/tasks/{taskId}`

响应：

```json
{
  "taskId": "task_xxx",
  "taskKind": "x.media_upload",
  "state": "running",
  "phase": "append",
  "progress": 0.42,
  "resultRef": null,
  "errorCode": null,
  "errorMessage": null
}
```

#### 7.1.6 获取任务结果

`GET /api/v1/tasks/{taskId}/result`

#### 7.1.7 取消任务

`POST /api/v1/tasks/{taskId}/cancel`

### 7.2 WebSocket 控制协议

#### 7.2.1 启动任务请求

消息类型：

- `request.start_task`

payload：

```json
{
  "taskId": "task_xxx",
  "taskKind": "x.media_upload",
  "inputRef": "task-store://task_xxx",
  "params": {
    "mimeType": "video/mp4",
    "totalBytes": 144960958,
    "tabId": 305047737
  }
}
```

#### 7.2.2 进度事件

消息类型：

- `event.task_progress`

payload：

```json
{
  "taskId": "task_xxx",
  "state": "running",
  "phase": "append",
  "progress": 0.26
}
```

#### 7.2.3 失败事件

消息类型：

- `event.task_failed`

payload：

```json
{
  "taskId": "task_xxx",
  "state": "failed",
  "phase": "append",
  "errorCode": "NETWORK_ERROR",
  "errorMessage": "net::ERR_HTTP2_PING_FAILED"
}
```

#### 7.2.4 完成事件

消息类型：

- `event.task_completed`

payload：

```json
{
  "taskId": "task_xxx",
  "state": "completed",
  "phase": "done",
  "resultRef": "task-result://task_xxx"
}
```

### 7.3 插件数据面读取协议

插件不得通过 websocket 或扩展控制消息传输大输入原文。  
统一改为由 background 通过本地 HTTP 接口拉取任务输入。

推荐接口：

`GET /api/v1/tasks/{taskId}/input/{partIndex}`

这样 plugin runtime 不再承担控制面之外的大对象传输职责。

## 8. WebSocket 层彻底重构要求

### 8.1 单写协程模型

每个 `ClientSession` 必须具备：

- `sendQueue chan []byte`
- `writerLoop goroutine`

写入流程必须统一为：

1. 业务方构造消息
2. 消息编码为 `[]byte`
3. 投递到 session `sendQueue`
4. `writerLoop` 串行调用 `Conn.WriteMessage`

任何模块不得直接：

- `sess.Conn.WriteMessage(...)`
- `conn.WriteMessage(...)`

### 8.2 读写职责分离

- `readerLoop` 只负责 `ReadMessage`
- `writerLoop` 只负责 `WriteMessage`
- 会话关闭由 session manager 统一处理

### 8.3 心跳语义

心跳必须调整为：

- 任意入站消息都刷新 `LastSeenAt`
- `ping/pong` 只是健康探针，不再是唯一存活依据
- session 关闭时，所有挂起 invocation 必须立即失败

### 8.4 挂起调用清理

当 session 断开时：

- 所有关联到该 session 的 pending callback 必须立即移除
- 对应 REST 请求必须立即返回错误
- 不允许继续等到固定超时

这条是必须实现的，否则 CLI 仍然会出现“看起来卡死”。

## 9. 插件长任务执行器重构要求

### 9.1 background 新职责

background 中新增 `TaskExecutor`：

- 接收 `start_task`
- 维护运行中的任务 map
- 拉取输入数据
- 驱动插件内部业务执行器
- 上报进度事件

### 9.2 业务执行器新职责

业务执行器层收敛为纯执行器：

- `prepare()`
- `run()`
- `reportProgress()`
- `complete()`

不得再承担：

- 本地任务元数据管理
- 跨多阶段恢复协调

### 9.3 错误分层

插件侧错误必须统一转为以下层级：

- `BRIDGE_ERROR`
- `INPUT_FETCH_ERROR`
- `TASK_PREPARE_ERROR`
- `TASK_EXECUTION_ERROR`
- `NETWORK_ERROR`
- `AUTH_ERROR`
- `TASK_CANCELLED`

CLI 和 REST 看到的错误必须直接来自这一层，不再拼接误导性文案。

## 10. 数据存储与资源清理

### 10.1 Task Data Store 存储位置

task data store 使用本地临时目录，例如：

```text
~/Library/Application Support/AIHub/tasks/{taskId}/input/
```

每个输入 part 单独存储，命名规则：

```text
part_000000
part_000001
...
```

### 10.2 清理规则

以下时机必须清理任务目录：

- `completed`
- `failed`
- `cancelled`
- `created` 后长期未完成输入
- `ready` 后长期未 `start`

### 10.3 清理责任

清理由 Go `Long Task Manager` 统一负责。  
插件不负责清理本地 task data store 目录。

## 12. 实施顺序

### 阶段 1：先重做 Go websocket 发送模型

目标：

- 单 writer loop
- 断连即失败 pending requests
- 消除并发写 panic

### 阶段 2：引入通用 Long Task Manager 和 Task Data Store

目标：

- 长任务任务化
- 大输入进入通用 task data store
- 完整移除同步桥接式长任务接口

### 阶段 3：插件改成任务执行器

目标：

- background 通过本地 REST 拉输入数据
- 插件业务执行器只做业务执行
- WS 只传控制和状态

### 阶段 4：CLI 改成任务式调用

目标：

- 创建任务
- 写入输入
- seal
- start
- poll
- fetch result

### 阶段 5：删除旧路径

目标：

- 删除所有同步桥接式长任务路径
- 删除所有控制面大 payload 传输路径
- 删除旧长任务兼容逻辑

## 13. 验收标准

### 13.1 平台功能验收

- 平台可以承载长任务
- 大输入不再受扩展控制消息上限影响
- 长任务执行期间心跳不中断
- session 断线时任务状态正确转失败或取消

### 13.2 架构验收

- WebSocket 层无任何直接并发写入口
- 上传数据不经过 websocket 传输
- 长任务不依赖同步 HTTP 长连接等待最终结果
- session 断开时挂起调用立即失败

### 13.3 可观测性验收

- CLI 可查询任务状态
- 可明确看到 `state / phase / progress`
- 错误包含明确 `errorCode` 和 `errorMessage`

### 13.4 代码验收

- 删除旧同步长任务实现
- 删除旧控制面大 payload 传输逻辑
- 所有长任务测试基于新任务模型通过

### 13.5 视频上传场景验收

视频上传只是第一批验证用例，必须满足：

- `a.mp4` 可稳定上传并发帖
- `b.mov` 可稳定上传并发帖
- 视频上传路径完全运行在通用长任务框架之上，而不是专用旁路实现

---

## 14. 最终结论

当前问题的根因不是单个”视频上传 API”细节错误，而是 LocalBridge 对长任务的承载模型设计错误。

必须同时完成以下两件事，问题才算被真正解决：

1. 重做 Go WebSocket 写模型，建立单 writer loop
2. 建立通用长任务模型，彻底分离控制面与数据面

只修其中一半都不够。

视频上传不是架构中心，只是第一个被纳入该模型的典型长任务。  
后续实现应严格以该通用架构为准，不再围绕单一业务场景设计桥接层。

## 15. 二期工程：事件推送优化

**状态：规划中**  
**优先级：P1 完成后启动**

### 15.1 背景

一期方案采用 REST 轮询模式查询任务状态，虽然简单可靠，但存在以下问题：

- 轮询产生大量无效请求
- 状态变化通知延迟（取决于轮询间隔）
- 客户端需要持续占用资源进行轮询

二期工程通过引入 WebSocket 事件推送机制，在保持一期架构稳定性的前提下，提升用户体验和系统效率。

### 15.2 设计原则

- **保持一期架构不变**：REST 仍然是控制面主要接口
- **事件推送作为增强**：WebSocket 推送是可选的优化路径
- **优雅降级**：WebSocket 断开时自动降级到轮询
- **向后兼容**：不影响已有的 REST 调用方式

### 15.3 混合架构

```text
Python CLI
  -> REST create/start task (控制操作)
  -> WebSocket subscribe to task events (可选，接收实时推送)
  -> REST fetch result when notified (按需拉取)
  
  降级路径：
  -> 如果 WebSocket 不可用，自动切换到 REST 轮询
```

### 15.4 新增协议

#### 15.4.1 WebSocket 订阅协议

**订阅任务事件**

消息类型：`request.subscribe_task`

payload：

```json
{
  “taskId”: “task_xxx”
}
```

响应：

```json
{
  “ok”: true,
  “taskId”: “task_xxx”
}
```

**取消订阅**

消息类型：`request.unsubscribe_task`

payload：

```json
{
  “taskId”: “task_xxx”
}
```

#### 15.4.2 事件推送格式

事件推送复用一期已定义的事件类型：

- `event.task_progress`
- `event.task_failed`
- `event.task_completed`

推送目标：所有订阅了该 `taskId` 的 WebSocket 会话

### 15.5 Go 侧实现要点

#### 15.5.1 新增模块

**TaskEventBroadcaster**

负责：

- 维护 `taskId -> []sessionId` 订阅关系
- 接收任务状态变化通知
- 向所有订阅者推送事件
- 处理订阅/取消订阅请求

最小接口：

```go
type TaskEventBroadcaster interface {
    Subscribe(taskId string, sessionId string) error
    Unsubscribe(taskId string, sessionId string) error
    BroadcastEvent(taskId string, event TaskEvent) error
}
```

#### 15.5.2 集成点

在 `Long Task Manager` 状态变化时触发推送：

```go
func (tm *TaskManager) updateTaskState(taskId string, newState TaskState) {
    // 更新任务状态
    tm.store.UpdateState(taskId, newState)
    
    // 推送事件给订阅者

        TaskId: taskId,
        State: newState,
        Phase: task.Phase,

```

### 15.6 CLI 侧实现要点

#### 15.6.1 事件监听

```python
class TaskClient:
    def __init__(self, rest_client, ws_client):
        self.rest = rest_client
        self.ws = ws_client
        self.use_push = ws_client.is_connected()
    
    def wait_for_completion(self, task_id):
        if self.use_push:
            return self._wait_with_push(task_id)
        else:
            return self._wait_with_poll(task_id)
    
    def _wait_with_push(self, task_id):
        # 订阅事件
        self.ws.subscribe_task(task_id)
        
        # 等待完成事件
        event = self.ws.wait_for_event(
            lambda e: e['taskId'] == task_id and 
                     e['state'] in ['completed', 'failed']
        )
        
        # 拉取最终结果
        return self.rest.get_task_result(task_id)
    
    def _wait_with_poll(self, task_id):
        # 降级到轮询
        while True:
            task = self.rest.get_task_status(task_id)
            if task['state'] in ['completed', 'failed']:
                return self.rest.get_task_result(task_id)
            time.sleep(2)
```

#### 15.6.2 自动降级

```python

    self.use_push = False
    # 正在等待的任务自动切换到轮询模式
```
