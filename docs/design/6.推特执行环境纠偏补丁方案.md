## 1. 问题结论

当前长任务重构在“平台任务化”方向上是对的，但在 `x.media_upload` 这个具体业务上犯了一个边界错误：

- 任何与 X/Twitter API 的实际交互，都必须发生在 `content` 环境
- `background service worker` 不能承担 X API 执行器角色

这不是“优化建议”，而是当前方案对浏览器运行边界的误判。

## 3. 新补丁目标

本补丁只纠正 `x.media_upload` 的执行环境和数据流，不推翻已完成的 Go 长任务底座。

补丁目标：

1. Go 继续负责任务创建、输入存储、状态机、结果存储、WebSocket 控制面
2. `background` 只做任务协调、Go 数据下载、本地 session 缓存、content 调度、事件回传
3. `content` 才是 `x.media_upload` 的真实执行器
4. X API 的 `INIT / APPEND / FINALIZE / STATUS` 全部留在 `content`
5. `background -> content` 只传分片，不传整文件
6. `content` 先缓存自己的 session，再开始调用现有上传能力
7. 任务完成后由 `background` 统一向 Go 上传结果并发 `event.task_completed`

## 4. 新的正确链路

### 4.1 总链路

```text
CLI
  -> Go CreateTask
  -> Go WriteInputPart/Seal/Start
  -> WebSocket request.start_task -> background
  -> background 从 Go 下载任务输入到 background session cache
  -> background 分片转发到 content
  -> content 缓存到 content session cache
  -> content 调用 X API 上传
  -> content -> background 回传 mediaId / progress / failed / cancelled
  -> background POST /result 到 Go
  -> background 通过 WebSocket 发 event.task_completed / failed / cancelled
  -> Go 收敛任务状态
```

### 4.2 关键边界

- Go 不理解 X 上传细节
- background 不调用 X API
- content 不直接持有 WebSocket
- `resultRef` 仍由 Go 生成

## 5. session 设计

### 5.1 background session

新增或泛化现有 `uploadSessions`：

- 归属：`background`
- 作用：缓存从 Go 下载完成的任务输入
- 生命周期：任务结束、失败、取消、浏览器关闭时释放
- 建议结构：

```ts
interface BackgroundTaskSession {
  taskId: string;
  mimeType: string;
  totalBytes: number;
  totalParts: number;
  chunks: string[]; // base64 chunks for content transport
  createdAt: number;
}
```

说明：
- 这里的 `chunks` 指的是 `background -> content` 传输块
- 不等于 CLI 上传到 Go 的原始 part
- 传输块大小固定为 `30 MiB raw equivalent`

### 5.2 content session

新增 content 本地 session：

- 归属：`content`
- 作用：接收 background 分片并缓存，随后在 content 环境内执行上传
- 生命周期：上传完成、失败、取消时释放
- 建议结构：

```ts
interface ContentUploadSession {
  taskId: string;
  mimeType: string;
  totalBytes: number;
  chunks: Blob[];
  createdAt: number;
}
```

说明：
- `content` 和 `background` session 不能共用
- `content` 必须先接收完成自己的 session，再开始上传

## 6. 传输策略

### 6.1 Go -> background

保留现有 Go 数据面接口：

- `GET /api/v1/tasks/{taskId}/input/{partIndex}`

但执行主体从“content 读取”改成“background 读取”。

`background` 在收到 `request.start_task` 后：

1. 读取任务元数据
2. 逐个从 Go 拉取原始 part
3. 按 `30 MiB raw equivalent` 重新切成传输块
4. 写入 `BackgroundTaskSession`

### 6.2 background -> content

保留并泛化现有 runtime message 机制：

- `GET_UPLOAD_SESSION_CHUNK`
- `RELEASE_UPLOAD_SESSION`

新增任务语义消息：

- `START_TASK_UPLOAD_FROM_BG_SESSION`
- `TASK_PROGRESS_FROM_CONTENT`
- `TASK_COMPLETED_FROM_CONTENT`
- `TASK_FAILED_FROM_CONTENT`
- `TASK_CANCELLED_FROM_CONTENT`
- `CANCEL_CONTENT_TASK`

传输规则：

- 单次 message 只传最多 `30 MiB raw equivalent`
- 使用 base64 string 作为跨环境载荷
- `content` 收到后立刻转成 `Blob` 存入自己的 session

### 6.3 content -> X API

上传开始前提：

- content session 已完整接收

上传策略：

1. `<= 64 MiB`  
   - 可直接复用 [twitter_api.ts](/Users/hyperorchid/aiwithblockchain/aihub/tweetClaw/src/x_api/twitter_api.ts) 的单文件上传能力

2. `> 64 MiB`  
   - 在 `content` 内从 content session 逐块调用现有 `INIT / APPEND / FINALIZE / STATUS`
   - 分块只发生在 content 内，不再依赖 background 直接执行 X API

## 10. 验收标准

补丁完成后，`x.media_upload` 必须满足：

1. Go 不直接参与 X API 执行
2. background 不直接调用 `fetch(upload.x.com)`
3. 所有 X API 上传动作都发生在 content 环境
4. `background -> content` 单条消息不超过 30 MiB raw equivalent
5. 50 MiB 视频继续可上传
6. 64 MiB 以上视频走分片传输并可上传
7. 任务完成后，仍通过 Go `resultRef + event.task_completed` 收敛
8. 浏览器关闭后 session cache 自动释放

## 11. 最终结论

这次补丁不是否定“任务化”本身，而是纠正“谁来执行 X API”这个最关键的运行边界。

可保留的部分：
- Go task model
- Go websocket model
- Go result store
- CLI task client方向

必须推翻的部分：
- `background` 作为 `x.media_upload` 执行器
- `content` 直接从 Go 拉输入并在 background 完成上传的 task3 设计

对 `x.media_upload` 来说，正确的角色划分只有一种：

- Go：任务平台
- background：任务协调器 + 本地 session cache
- content：X API 执行器
