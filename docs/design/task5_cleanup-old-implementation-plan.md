## 1. 计划概述

本阶段目标是删除所有旧的同步桥接式长任务实现，完成从旧架构到新架构的彻底迁移。

阶段 1-4 已完成新架构的建立和业务迁移：
- 阶段 1：WebSocket 单写模型已建立
- 阶段 2：Long Task Manager 和 Task Data Store 已实现
- 阶段 3：插件已改造为任务协调器模式
- 阶段 4：CLI 已改为任务式调用

阶段 5 在此基础上，彻底清理旧代码，确保系统中不再存在任何旧的长任务实现路径。

---

## 2. 本阶段目标

### 2.1 必须达成的结果

阶段 5 完成后，系统必须满足：

1. 所有旧的同步桥接式长任务接口已删除
2. 所有控制面大 payload 传输逻辑已删除
3. 所有旧的视频上传实现已删除
4. 代码库中不存在任何"单个 HTTP 请求等待整个执行完成"的长任务模型
5. 文档已更新，反映新架构
6. 所有测试基于新任务模型通过

### 2.2 本阶段不做的事情

本阶段不引入新功能，不优化性能，不重构已迁移的新代码。

本阶段的唯一职责是删除旧实现，确保迁移的彻底性和完整性。

---

## 3. 核心设计原则

### 3.1 彻底性原则

删除必须彻底：

- 不保留"以防万一"的旧代码
- 不保留注释掉的旧实现
- 不保留"兼容旧客户端"的分支逻辑
- 不保留未使用的旧类型定义

### 3.2 安全性原则

删除必须安全：

- 删除前必须确认新实现已完全替代
- 删除前必须确认所有测试通过
- 删除后必须验证编译通过
- 删除后必须验证集成测试通过

### 3.3 可追溯性原则

删除必须可追溯：

- 每次删除必须有明确的 commit message
- commit message 必须说明删除的是什么、为什么删除
- 重要删除必须在文档中记录

## 6. 风险和缓解措施

### 风险 1：误删仍在使用的代码

**风险描述**：可能误删看似旧实现但实际仍在使用的代码

**缓解措施**：
- 删除前仔细审查代码引用
- 使用 IDE 的"查找引用"功能
- 删除后立即运行完整测试套件
- 分批删除，每批后验证

### 风险 2：新实现未完全覆盖旧功能

**风险描述**：旧实现可能包含新实现未覆盖的边缘场景

**缓解措施**：
- 删除前对比新旧实现的功能覆盖
- 补充缺失的测试用例
- 保留 git 历史，必要时可恢复
- 分阶段删除，逐步验证

### 风险 3：文档更新不完整

**风险描述**：可能遗漏某些文档中的旧实现引用

**缓解措施**：
- 使用全局搜索查找旧接口名称
- 搜索旧的 URL 路径
- 搜索旧的类型名称
- 团队交叉审查文档

### 风险 4：测试覆盖不足

**风险描述**：删除后可能暴露测试覆盖的盲区

**缓解措施**：
- 删除前确保新实现测试覆盖完整
- 补充端到端测试
- 补充集成测试
- 进行手工测试验证
